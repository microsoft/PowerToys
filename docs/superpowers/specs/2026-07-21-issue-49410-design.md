## Summary

PowerDisplay will support changing brightness by scrolling while the pointer is over its existing notification-area icon. A new three-state setting will control all PowerDisplay mouse-wheel behavior:

1. `Disabled`
2. `PrimaryDisplay`
3. `AllDisplays`

`PrimaryDisplay` is the default, including for settings files created before this feature. The selected display target applies only to tray-icon scrolling. When the mode is enabled, the existing flyout sliders continue to react to the mouse wheel. When the mode is disabled, both tray-icon scrolling and flyout-slider scrolling are disabled.

The implementation will use Explorer's icon-specific mouse-move callback to confirm that the pointer reached the real PowerDisplay icon, `Shell_NotifyIconGetRect` to cache and revalidate its screen rectangle, and an on-demand `WH_MOUSE_LL` hook on a dedicated message-loop thread to obtain the signed wheel delta. The hook will be installed only while the icon is hovered and will never consume the input.

## Goals

- Adjust brightness without opening the PowerDisplay flyout.
- Reuse the existing configurable mouse-wheel increment.
- Let users disable all PowerDisplay mouse-wheel adjustment.
- Let users choose whether tray scrolling adjusts the primary display or all visible controllable displays.
- Preserve each display's existing brightness offset in all-display mode.
- Work for both pinned and overflow notification-area icons.
- Keep the global hook scoped, fast, and independent of UI or hardware work.
- Preserve existing tray click, context-menu, settings-update, and Explorer-restart behavior.
- Preserve backward compatibility for existing settings files.

## Non-goals

- A brightness OSD or toast.
- A foreground-window, cursor-monitor, or per-monitor tray target.
- Changing linked-brightness exclusions or semantics inside the flyout.
- Runner, GPO, installer, CLI, or PowerDisplay IPC changes.
- Consuming the wheel event or preventing other applications and hooks from seeing it.
- Controlling displays that the user has hidden in PowerDisplay.

## Confirmed Product Decisions

- The setting has three options: off, primary display, and all displays.
- The default is primary display.
- Off disables both tray-icon wheel adjustment and flyout-slider wheel adjustment.
- All-display mode applies the same relative delta to each display's own current brightness, preserving differences between displays.
- Hidden displays are excluded.
- The tray target setting takes precedence over linked-brightness mode.
- Tray scrolling may temporarily make displays differ while linked-brightness mode is enabled. The next user change to the flyout's All Displays slider resynchronizes them through the existing linked-brightness path.
- No OSD is added in the first implementation.

## Current Architecture

`PowerDisplay/Helpers/TrayIconService.cs` owns the hidden WinUI window, notification icon registration, tray callback message, context menu, Explorer restart handling, and icon teardown.

`PowerDisplayXAML/App.xaml.cs` creates `MainWindow`, then creates `TrayIconService`. The existing settings-updated named event first calls `MainViewModel.ApplySettingsFromUI()` and then calls `TrayIconService.SetupTrayIcon()`.

`MainViewModel.LoadUIDisplaySettings()` loads `MouseWheelIncrement` and the other flyout settings. Individual monitor controls expose values through `MonitorViewModel`.

`SliderExtensions` implements the current flyout wheel behavior through the `IsMouseWheelEnabled` and `MouseWheelChange` attached properties.

`MonitorViewModel.Brightness` updates the in-memory value immediately and schedules the existing debounced hardware write. This is the correct write path to reuse for tray scrolling.

## Approaches Considered

### 1. On-demand low-level mouse hook

Explorer confirms hover over the PowerDisplay icon. PowerDisplay then installs `WH_MOUSE_LL`, reads `MSLLHOOKSTRUCT.mouseData`, and removes the hook when the pointer leaves.

This is the selected approach because it receives the system's processed `WM_MOUSEWHEEL` semantics, provides the event's screen point and signed delta, has precedent in PowerToys, and can be tightly scoped to icon hover.

### 2. Raw Input

PowerDisplay could register the hidden window for background mouse input and parse `WM_INPUT`.

This avoids the low-level hook timeout, but raw-input mouse registration is process-wide, reports device-level packets, requires additional leave detection, and introduces more interop and lifecycle complexity. It is not a clear behavioral improvement for this narrowly scoped interaction.

### 3. Runner input broker

Runner could capture mouse input and forward tray wheel events to PowerDisplay.

This would add a cross-process contract and touch shared lifecycle code for a feature owned entirely by PowerDisplay. The scope and compatibility cost are not justified.

## Settings Model

Add a shared `MouseWheelControlMode` enum in `PowerDisplay.Models`, which is already
referenced by both `Settings.UI.Library` and `PowerDisplay.Lib`:

```csharp
public enum MouseWheelControlMode
{
    Disabled = 0,
    PrimaryDisplay = 1,
    AllDisplays = 2,
}
```

Add `PowerDisplayProperties.MouseWheelControlMode` with JSON name `mouse_wheel_control_mode`.

The `PowerDisplayProperties` constructor explicitly initializes the value to `PrimaryDisplay`. `System.Text.Json` calls this constructor before populating fields, so legacy JSON without the property keeps the chosen default without a separate migration.

Runtime consumers must normalize unsupported numeric enum values to `Disabled`. A missing property and an unsupported property are intentionally different:

- Missing means the user has not made a choice and receives the product default, `PrimaryDisplay`.
- Unsupported or corrupt means the binary cannot safely interpret the requested behavior and performs no mouse-wheel hardware action.

The Settings UI will add a localized combo box before the existing mouse-wheel increment card, with options displayed in this order:

1. Off
2. Primary display
3. All displays

The increment card is disabled when the selected mode is `Disabled`. Changing either the mode or increment signals the existing `SettingsUpdatedPowerDisplayEvent`, so the running module updates immediately.

## Flyout Behavior

`MainViewModel` will expose:

- `MouseWheelControlMode`
- `IsMouseWheelControlEnabled`

`MonitorViewModel` will expose the parent's enabled state in the same way it currently exposes `MouseWheelIncrement`.

The four flyout sliders currently set `SliderExtensions.IsMouseWheelEnabled="True"`. They will instead bind that attached property to the applicable view-model enabled state:

- The linked All Displays brightness slider binds to `MainViewModel.IsMouseWheelControlEnabled`.
- Per-monitor brightness, contrast, and volume sliders bind to `MonitorViewModel.IsMouseWheelControlEnabled`.

When the mode changes, existing monitor view models receive property-change notifications for both the enabled state and increment.

Disabling the attached property removes the `PointerWheelChanged` handler. The wheel event is therefore not marked handled and can continue to scroll the flyout's containing `ScrollViewer`. Setting the increment to zero is not an acceptable substitute because `SliderExtensions` treats non-positive values as a request to use `Slider.SmallChange`.

## Tray Hover Detection

The implementation must not use a fixed taskbar rectangle or rely only on `Shell_NotifyIconGetRect`.

`TrayIconService` already receives the application-defined `WmTrayIcon` callback. With the current notification-icon version, Explorer places the icon ID in `wParam` and the associated mouse message in `lParam`. Explorer sends `WM_MOUSEMOVE` only when the pointer moves through this specific icon.

On an icon-specific `WM_MOUSEMOVE` callback:

1. Verify that mouse-wheel control is enabled.
2. Call `GetCursorPos`.
3. If a fresh cached rectangle already contains the point, keep the current generation armed without another Shell call.
4. Otherwise, build `NOTIFYICONIDENTIFIER` from the existing hidden-window handle and `MyNotifyId`.
5. Call `Shell_NotifyIconGetRect` on the UI thread and require the current point to be inside the returned rectangle.
6. Cache the screen rectangle, advance a hover generation token, and arm the mouse-wheel listener.

This two-signal design avoids the notification-chevron ambiguity. Windows can return the chevron rectangle when an icon is hidden in the overflow area, but hovering the chevron does not produce a mouse-move callback for the PowerDisplay icon and therefore cannot arm the listener. When the overflow panel is open and the real icon is hovered, Explorer produces the icon-specific callback.

Both the returned `RECT` and `MSLLHOOKSTRUCT.pt` use screen coordinates, so no XAML or per-monitor DPI conversion is required.

## Mouse Listener Lifecycle

`TrayIconMouseWheelListener` will own a dedicated thread with a Win32 message loop. The thread may live for the tray service lifetime, but the global hook does not.

The listener has the following states:

- `Disabled`: no hook can be installed.
- `Idle`: enabled, but the icon has not been confirmed as hovered.
- `Armed`: Explorer confirmed hover, a valid rectangle is cached, and the hook is installed.
- `Stopping`: teardown has been requested and no further samples are accepted.

State transitions:

1. Successful `NIM_ADD` and enabled mode: `Disabled` to `Idle`.
2. Icon-specific `WM_MOUSEMOVE` with a valid rectangle: `Idle` to `Armed`.
3. Pointer movement outside the cached rectangle: `Armed` to `Idle`, then unhook after the current callback returns.
4. Bounds revalidation failure or stale hover generation: `Armed` to `Idle`.
5. Mode disabled, icon hidden, Explorer restart, tray destruction, or module exit: any state to `Disabled` or `Stopping`, with synchronous hook release.

The low-level callback handles only `WM_MOUSEMOVE` and `WM_MOUSEWHEEL`:

- For `nCode < 0`, immediately call `CallNextHookEx`.
- For mouse movement, compare the screen point with the cached rectangle and request disarm when it leaves.
- For wheel input, copy the point, timestamp, signed high word of `mouseData`, and hover generation into a small sample, then enqueue it for UI-thread processing.
- Always call and return `CallNextHookEx`; never return a consuming result.

The callback must not call Shell APIs, allocate unbounded work, use LINQ, log, touch WinUI objects, inspect monitors, or perform hardware I/O. The dedicated thread prevents a busy UI thread from causing the hook to exceed `LowLevelHooksTimeout` and be silently removed by Windows.

The listener must use AOT-compatible interop: `LibraryImport`, blittable native structures, and a callback whose lifetime is explicitly rooted or represented by an unmanaged function pointer.

## Wheel Sample Validation and Accumulation

The UI-thread handler drains queued samples in generation-based batches. It queries the
current icon rectangle once per batch, then revalidates every sample in that batch:

1. The mode is not `Disabled`.
2. PowerDisplay is initialized and `IsInteractionEnabled` is true.
3. The sample's hover generation is still current.
4. `Shell_NotifyIconGetRect` succeeds for the current icon identity.
5. The sample's screen point is inside the current rectangle.

Only validated samples enter `WheelDeltaAccumulator`.

The accumulator:

- Adds signed wheel deltas.
- Emits one notch for each complete `WHEEL_DELTA` unit of 120.
- Preserves a partial remainder for high-resolution wheels and touchpads.
- Naturally cancels partial movement when the direction reverses.
- Resets when hover ends, the mode or target changes, the icon is recreated, or validation fails.

The adjustment amount is computed with widened arithmetic:

```text
delta = (long)notches * MouseWheelIncrement
```

This prevents overflow before each target value is clamped to 0 through 100.

## Display Target Selection

`MainViewModel.AdjustBrightnessFromTrayWheel` delegates target selection and value planning to a pure helper.

### Primary display

`MonitorNumber == 1` must not be used. PowerDisplay currently assigns monitor numbers from the `QueryDisplayConfig` path index, which is not a primary-display flag.

Resolve the primary logical display through Win32:

1. Call `MonitorFromPoint({0, 0}, MONITOR_DEFAULTTOPRIMARY)`.
2. Call `GetMonitorInfoW` for the resulting `HMONITOR`.
3. Read `MONITORINFOEX.szDevice`.
4. Match that GDI device name against monitor view models.

`MonitorViewModel` will expose its underlying monitor's `GdiDeviceName`.

In mirror mode, more than one physical display can share the primary logical source's GDI device name. All matching physical displays that otherwise qualify are adjusted.

If PowerDisplay cannot resolve a primary GDI name or no valid target matches it, the event performs no hardware write. It must not fall back to an arbitrary monitor.

### All displays

Select every item in `MainViewModel.Monitors` that:

- Is visible to PowerDisplay. Hidden monitors are already removed from this collection.
- Supports brightness.
- Has a trustworthy current value, indicated by `MonitorReadFlags.Brightness`.

The same relative delta is applied independently to each selected display:

```text
newBrightness = clamp(currentBrightness + delta, 0, 100)
```

This preserves brightness differences. A target with an unknown current brightness is skipped because relative adjustment from the model's default value would create an unintended absolute write. Other valid all-display targets still proceed.

## Linked Brightness Interaction

Tray scrolling deliberately does not consult:

- `LinkedLevelsActive`
- `ExcludedFromSyncMonitorIds`
- The current `LinkedBrightness` master value

The planner directly updates the selected monitor view models. This means primary-display mode can temporarily diverge displays while linked mode remains enabled, and all-display mode can preserve preexisting differences.

The flyout's linked mode remains a separate interaction mode. The next user change to the All Displays slider uses the existing linked-brightness broadcast and resynchronizes its linked targets. Tray scrolling must not silently disable linked mode or rewrite its exclusion list.

## Hardware Write Path

For each planned adjustment, `MainViewModel` assigns the corresponding `MonitorViewModel.Brightness` property.

This reuses:

- Immediate in-memory and UI updates.
- Per-monitor clamping safeguards.
- The existing 200 ms debounce.
- `MonitorManager`'s WMI and DDC/CI routing.
- Existing write-failure logging.

No monitor API call occurs from the hook or tray service.

## Error Handling

- Hook-thread creation or hook installation failure logs a rate-limited warning and leaves tray click and flyout behavior available.
- `Shell_NotifyIconGetRect` failure invalidates the hover generation and disarms the listener.
- Failure to enqueue a sample drops that sample and does not block the hook.
- Unsupported settings values use effective mode `Disabled`.
- Missing or unread brightness values are never treated as zero.
- No target is a safe no-op with a rate-limited diagnostic outside the hook path.
- Hardware failures continue through the existing `MonitorViewModel` and `MonitorManager` error paths.
- Teardown is idempotent so settings updates, Explorer restart, window close, and process exit can safely converge on the same cleanup path.

Expected transient conditions must not produce one log entry per wheel packet.

## Localization, Accessibility, and Telemetry

Add English resource keys for:

- Mouse wheel control
- Mouse wheel control description
- Off
- Primary display
- All displays

The combo box must have a localized accessible name and description through the existing `x:Uid` pattern. The setting uses standard Settings UI controls and requires no custom keyboard interaction.

No telemetry is required for the first implementation. Existing settings telemetry does not currently report the mouse-wheel increment or linked-brightness setting, and adding a new event field is not necessary to deliver or validate the feature.

## Acceptance Criteria

1. The Settings UI exposes Off, Primary display, and All displays.
2. Primary display is the default for new and legacy settings.
3. Off installs no active mouse hook and disables wheel adjustment on all PowerDisplay flyout sliders.
4. When enabled, moving over the real PowerDisplay icon arms the listener; the overflow chevron alone does not.
5. Scrolling over the icon adjusts brightness by the configured increment without opening the flyout.
6. Primary mode adjusts only valid physical displays associated with the primary logical display.
7. All mode applies the same relative delta to every visible valid display and preserves differences.
8. Tray behavior remains independent of linked-brightness mode and exclusions.
9. High-resolution wheel input accumulates correctly.
10. Every input continues through `CallNextHookEx`.
11. No Shell, WinUI, logging, monitor-selection, or hardware work runs in the hook callback.
12. Hook and cached hover state are released on leave, disable, icon removal, Explorer restart, and process exit.
13. Existing tray click, context menu, flyout controls, settings updates, and debounced hardware writes continue to work.

## Risks and Mitigations

### Low-level hook removal

Windows silently removes a hook that exceeds `LowLevelHooksTimeout`.

Mitigation: use a dedicated message-loop thread, keep the callback constant-time, and dispatch all meaningful work elsewhere.

### Overflow icon ambiguity

`Shell_NotifyIconGetRect` can identify the overflow chevron when the icon itself is hidden.

Mitigation: never arm from the rectangle alone. Require Explorer's callback for the PowerDisplay icon before accepting the rectangle.

### Stale display topology

The primary display or monitor list can change while wheel samples are queued.

Mitigation: reject input while scanning and resolve the primary GDI device on the UI thread at application time.

### Unknown brightness values

A supported display can fail its initial read, leaving a fabricated default value.

Mitigation: require `MonitorReadFlags.Brightness` before relative adjustment.

### Linked-mode visual mismatch

Tray adjustment can temporarily diverge values while the linked flyout remains enabled.

Mitigation: this is an explicit product decision. Do not rewrite the linked setting; the next linked-slider interaction resynchronizes displays.

### Rapid hardware writes

High-resolution wheels can produce many events.

Mitigation: accumulate partial deltas and reuse the existing per-monitor debounce.
